# 全链路学习助手：需求分析与产品设计 V1

## 1. 背景与产品判断

当前项目已经完成基础部署，是一个纯静态 Web 学习操作台，面向在职产品经理与研发管理者，帮助用户在 12 周内建立 Git/GitHub、AI 辅助编程、研发协作、CI/CD、发布复盘等全链路能力。

这个产品不应被设计成普通课程站点。它的核心价值不是“展示学习内容”，而是把学习过程变成一个可执行、可追踪、可复盘、可展示的个人成长项目。

一句话定位：

> 面向产品经理和研发管理者的全链路技术协作学习工作台，帮助用户用每周约 5 小时，从看懂研发协作到独立推进一个 AI 辅助小产品闭环。

## 2. 目标用户

### 核心用户

1. 在职产品经理
   - 已经参与需求、评审、验收、上线，但对 Git、PR、CI/CD、发布链路缺乏系统理解。
   - 希望能更专业地和研发沟通，而不是停留在“提需求、催进度、看效果”。

2. 研发管理者或技术协作型负责人
   - 不一定每天写代码，但需要理解研发流程、工程质量、分支协作、发布风险。
   - 希望把 AI 工具纳入团队协作流程，提高需求拆解、评审和排障效率。

3. 想从业务/产品转向 AI 产品、技术产品、研发管理的人
   - 有真实工作场景，但缺少把工具链串起来的训练路径。

### 非目标用户

- 零基础且只想学习编程语法的用户。
- 追求系统计算机课程、算法、后端架构深度训练的用户。
- 需要多人班级管理、作业批改、付费课程体系的机构用户。

## 3. 用户痛点

1. 学习路径碎片化
   - Git、GitHub、AI 编程、CI/CD、发布各自都有教程，但很少围绕产品经理的真实工作流组织。

2. 学完不会用
   - 很多学习停留在视频观看，缺少可执行任务、真实产出和复盘机制。

3. 和研发沟通缺少抓手
   - 看不懂 PR，不知道冲突、分支、发布失败、安全风险意味着什么，也不知道该问哪些问题。

4. AI 使用容易失控
   - 用户会让 AI 写代码，但常常缺少上下文、约束、验收标准，导致改动范围扩大、无法判断正确性。

5. 进度和成果不可见
   - 学习没有沉淀到仓库、周报、复盘和最终作品里，难以证明能力提升。

## 4. 产品目标

### 用户目标

- 能在 12 周内完成一条可执行学习路线。
- 能用 GitHub 仓库承载学习记录、任务、PR、复盘。
- 能理解研发协作关键概念，并在工作中提出更专业的问题。
- 能用 AI 辅助完成一个小产品从需求到发布的闭环。
- 能形成可展示成果：项目地址、PR 记录、周报、复盘文章。

### 产品目标

- 降低开始学习的心理门槛。
- 让用户每次打开都知道“下一步做什么”。
- 把学习动作从“看内容”转为“做任务 + 留痕迹 + 复盘”。
- 在不引入后端的前提下，先跑通单人使用闭环。

## 5. 核心使用场景

### 场景 A：每周开始时规划学习

用户打开“今日面板”，看到当前周主题、完成进度、本周任务和建议节奏，选择本周最关键的 2-4 个动作。

关键体验：
- 不需要重新判断学习顺序。
- 清楚知道本周产出是什么。
- 学习负担控制在在职用户可接受范围。

### 场景 B：遇到 Git 或协作问题时自救

用户进入“Git 急救”，搜索冲突、撤销、PR、同步等场景，获得低风险操作步骤和协作解释。

关键体验：
- 先判断状态，再操作。
- 把技术命令翻译成产品和研发协作语言。
- 避免危险命令和无上下文操作。

### 场景 C：用 AI 辅助研发协作

用户进入“AI 提示词”，复制适合的提示词，用于 PRD 拆解、PR Review、代码解释、排障、发布检查、学习复盘。

关键体验：
- 提示词面向真实工作流，而不是泛泛提问。
- 输出结构可直接转为 Issue、PR 描述、检查表或复盘。
- 强调约束、验收、影响范围。

### 场景 D：阶段性完成闭环项目

用户在第 11-12 周选择一个真实小工具，完成问题定义、需求拆解、AI 协作实现、Git 流程、发布入口和复盘展示。

关键体验：
- 项目小而完整。
- 有可访问结果。
- 能证明用户具备产品判断、技术理解和 AI 协作能力。

### 场景 E：多端延续与成果导出

用户在电脑上学习，在另一台设备导入 JSON 继续；每周导出 Markdown 周报放进 GitHub 仓库。

关键体验：
- 当前无需账号体系也能使用。
- 进度不被锁在单次浏览器会话里。
- 输出物可以长期沉淀。

## 6. 现有功能盘点

已具备：

- 前端登录门禁。
- 今日面板：总体进度、当前主题、下一步行动、本周任务、周节奏。
- 12 周计划：每周主题、学习负载、目标产出、任务、笔记。
- Git 急救卡：常见场景搜索、命令参考。
- AI 提示词库：工作流提示词搜索、复制。
- 闭环项目：里程碑与验收标准。
- 多端入口：导入导出、周报、使用说明。
- localStorage 本地保存。
- PWA 基础文件。
- 纯静态部署和腾讯云公网访问记录。

主要短板：

- 用户首次进入时缺少“我是谁、该选哪条路线”的引导。
- 12 周计划目前是一条固定路线，缺少按用户基础动态调整。
- 任务完成只有勾选，缺少更明确的产出模板和示例。
- Git 急救和提示词库还不能沉淀用户自己的常用卡片。
- 多端同步依赖手动导入导出，长期使用成本较高。
- 登录门禁只是前端保护，不适合承载敏感数据。

## 7. 产品结构设计

### 信息架构

1. 今日面板
   - 当前周进度
   - 下一步行动
   - 本周任务
   - 本周节奏
   - 最近一次复盘

2. 12 周计划
   - 每周主题
   - 学习目标
   - 视频/材料
   - 任务清单
   - 产出模板
   - 笔记和复盘

3. Git 急救
   - 场景搜索
   - 风险提示
   - 推荐命令
   - 需要向研发确认的问题

4. AI 提示词
   - 场景分类
   - 提示词模板
   - 复制使用
   - 用户自定义模板

5. 闭环项目
   - 选题
   - PRD
   - Issue 拆解
   - PR 和 Review
   - 发布检查
   - 复盘展示

6. 多端与数据
   - 导出 JSON
   - 导入 JSON
   - 导出周报
   - 同步方案说明

### 导航原则

- 首页永远回答“我现在该做什么”。
- 学习计划回答“我为什么做这些，以及做到什么算完成”。
- 急救和提示词回答“我卡住时怎么继续推进”。
- 闭环项目回答“这 12 周最后能交付什么”。
- 多端入口回答“我的进度和成果如何带走”。

## 8. 功能优先级

### P0：保留并打磨现有单人学习闭环

1. 首次使用引导
   - 让用户选择身份：产品经理、研发管理者、AI 产品转型。
   - 让用户选择基础：完全不了解 Git、会用 GitHub 网页、能看懂简单代码。
   - 根据选择给出推荐开始周或提醒。

2. 每周产出模板
   - 为每周增加“本周最终产出”模板。
   - 例如概念地图、Issue 模板、PR 描述、发布检查表、复盘模板。

3. 更强的复盘区
   - 每周至少记录：完成了什么、卡点、工作中如何应用、下周行动。
   - 周报导出应包含复盘内容，而不只是完成状态。

4. 风险提示
   - 在 Git 急救中突出安全原则：先 `git status` / `git diff`，谨慎撤销和覆盖。
   - 对密钥、Token、部署、公开仓库增加明确提醒。

### P1：提高长期使用价值

1. 自定义卡片
   - 用户可以新增自己的 Git 急救卡和 AI 提示词。
   - 仍保存在本地或导出 JSON。

2. 学习资产库
   - 集中展示用户已经产出的 README、PRD、Issue、PR、周报、复盘链接。
   - 让学习成果从“勾选任务”变成“作品集”。

3. 项目选题向导
   - 通过几个问题帮助用户选择闭环项目。
   - 输出最小可行范围、非目标、验收标准。

4. 进度提醒
   - 不一定做复杂通知，先做“距离上次学习已过去 X 天”的轻提醒。

### P2：云同步和多人能力

1. 真账号体系
   - 可选 Cloudflare Access、Supabase Auth 或服务端认证。

2. 云端同步
   - Supabase、GitHub Gist 或 Notion API。
   - 需要明确隐私边界和数据结构。

3. 课程/训练营模式
   - 班级、学员、作业、导师反馈。
   - 这不是当前 V1 的重点，除非后续商业化目标明确。

## 9. 页面级设计建议

### 今日面板

目标：减少用户决策成本。

建议增加：
- “今天只做一件事”模块。
- 最近一次笔记摘要。
- 本周产出入口。
- 如果用户连续多周未完成，允许“调整节奏”，不要只显示落后。

### 12 周计划

目标：让每周任务和能力提升强绑定。

建议增加：
- 每周“完成定义”。
- 每周“可复制模板”。
- 每周“工作场景应用”。
- 每周“常见误区”。

### Git 急救

目标：让非研发用户安全地处理和描述问题。

建议增加：
- 每张卡增加风险级别：低、中、高。
- 增加“我应该向研发确认什么”。
- 高风险命令默认不提供，或只解释风险。

### AI 提示词

目标：让 AI 成为协作放大器，而不是无约束代写工具。

建议增加：
- 按场景分类：需求、代码、Git、Review、发布、复盘。
- 支持复制后自动标记最近使用。
- 增加提示词质量原则：上下文、目标、约束、验收、输出格式。

### 闭环项目

目标：把最终成果收敛成可展示案例。

建议增加：
- 选题评估表：真实频率、范围大小、数据复杂度、发布难度。
- PRD 模板。
- Issue 拆解模板。
- 发布检查表。
- 项目复盘模板。

## 10. 核心指标

如果只是个人工具，建议看这些指标：

- 12 周任务完成率。
- 每周复盘填写率。
- 周报导出次数。
- 最终闭环项目完成情况。
- 用户是否形成至少 3 个可复用工作资产：PRD、PR 模板、发布检查表、项目复盘。

如果后续要做公开产品，再补充：

- 首次访问到完成第一项任务的转化率。
- 第 7 天留存。
- 第 4 周留存。
- 模板复制次数。
- 导入/导出使用率。
- 付费或转介绍意愿。

## 11. V1 迭代路线建议

### 第 1 阶段：补齐产品闭环

- 新增首次使用引导。
- 每周增加产出模板和完成定义。
- 周报导出加入笔记和复盘。
- Git 急救卡增加风险提醒。

### 第 2 阶段：强化个人资产

- 支持自定义提示词和急救卡。
- 增加学习资产库。
- 增加闭环项目选题向导。

### 第 3 阶段：判断是否需要云端化

- 先观察是否真的高频多端使用。
- 如果只是个人使用，可继续 JSON 导入导出。
- 如果多人或长期使用，再接入真实认证和云同步。

## 12. 待澄清问题

这些问题会影响后续产品形态，建议优先确认：

1. 这个工具主要是你自己用，还是未来希望公开给其他产品经理使用？
2. 目标用户更偏“完全不懂代码的产品经理”，还是“有一点技术基础、想系统补齐协作链路的人”？
3. 12 周计划是否已经对应一套固定视频课程？如果是，视频标题和顺序是否需要保持强绑定？
4. 你希望最终产出是“个人学习工具”，还是“可包装成课程/训练营的产品”？
5. 当前登录门禁是否只是临时防误入，还是你希望它真的保护学习数据？
6. 是否需要把学习数据同步到手机、平板和多台电脑，还是导入导出暂时足够？
7. 闭环项目更希望偏产品管理工具、AI 小应用，还是 GitHub/研发协作工具？
8. 你希望产品调性更像“严肃工作台”，还是更像“陪伴式学习教练”？

## 13. 当前建议结论

建议短期不要急着做复杂后端、账号体系或多人功能。当前最值得做的是把“学习路线”升级为“学习产出系统”：

- 每周有明确产出。
- 每个产出有模板。
- 每次学习有复盘。
- 最后能组合成一个可展示项目案例。

这样即使产品仍然是纯静态页面，也已经能提供清晰价值。等个人使用路径跑顺，再决定是否做云同步、公开化和商业化。
