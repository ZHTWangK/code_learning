# 全链路学习助手：需求分析文档

版本：V1.0  
日期：2026-05-24  
项目阶段：基础部署后，进入个人学习工具产品化设计阶段

## 1. 项目背景

当前项目是一个已完成基础部署的 Web 应用，定位为“全链路学习助手”。现阶段它首先服务于个人长期学习、记录和能力建设，围绕 Git/GitHub、AI 辅助编程、研发协作、CI/CD、发布和复盘，提供一套可调整的 12 周学习与实践路径。

从现有产品形态看，它已经具备学习计划、任务勾选、笔记、Git 急救、AI 提示词、闭环项目、多端导入导出和前端登录门禁等能力。

本阶段的核心任务不是继续堆功能，而是先明确：

- 这个产品如何优先服务个人成长。
- 用户为什么需要它。
- 它和普通课程、笔记工具、项目管理工具有什么区别。
- 下一阶段应该优先补齐哪些能力。

## 2. 产品定位

一句话定位：

> 面向浅技术基础产品经理的陪伴式全链路学习教练，帮助用户围绕真实学习、记录、实践和复盘，逐步打造具备技术协作、AI 工具化和小产品交付能力的高竞争力个体。

产品关键词：

- 个人成长操作台
- 陪伴式学习教练
- 产品经理技术协作
- Git/GitHub 入门到协作
- AI 辅助编程与研发管理
- 小产品闭环交付
- 资源推荐与学习记录
- 可追踪、可复盘、可展示

产品不应被设计成普通课程站点。它的核心价值不是“展示学习资料”，而是把学习过程变成一个可执行、可追踪、可复盘、可展示的个人成长项目。半年到一年后，产品可以自然外化为 blog、公开工具或个人方法论展示页，但短期所有设计都应优先服务自用效率和长期沉淀。

## 3. 目标用户

### 3.1 当前核心用户

当前最核心用户是“有浅显开发概念的产品经理本人”。用户知道开发、前端、后端、测试、上线等大概概念，但没有系统掌握 GitHub 协作、AI 辅助编程、CI/CD 和完整交付链路。

这个产品更像个人学习教练和成长档案，而不是通用 SaaS。它服务的长期目标是打造一个更有竞争力的个体，为未来一人公司、个人品牌、独立工具产品、财务自由等方向积累能力和资产。

### 3.2 未来可扩展用户

#### 用户 A：在职产品经理

典型特征：

- 参与需求、评审、排期、验收和上线。
- 有很浅显的开发概念，经常和研发协作，但对 Git、PR、CI/CD、发布链路缺乏系统理解。
- 希望自己能更专业地判断研发风险和沟通问题。

核心诉求：

- 看懂研发在做什么。
- 看懂 PR、分支、冲突、发布失败等常见协作问题。
- 能把 PRD 拆成更工程化的任务。
- 会用 AI 辅助需求拆解、代码理解、排障和复盘。

#### 用户 B：研发管理者或技术协作负责人

典型特征：

- 不一定每天写代码，但需要理解研发流程和工程质量。
- 关注团队协作、质量门禁、发布风险、权限和安全。

核心诉求：

- 系统梳理研发协作链路。
- 把 GitHub、PR、CI/CD、发布检查转化为管理动作。
- 建立可复用的协作模板和检查清单。

#### 用户 C：想转向 AI 产品或技术产品的人

典型特征：

- 有产品或业务经验。
- 想补齐技术理解力，避免只停留在概念层。
- 希望产出一个可展示的小作品。

核心诉求：

- 用低门槛路径补齐技术协作能力。
- 通过一个完整小产品证明自己理解 AI 与研发交付。

### 3.3 非目标用户

当前阶段不优先服务：

- 只想系统学习编程语法的零基础编程用户。
- 需要算法、计算机基础、后端架构深度课程的用户。
- 当前就需要班级、助教、作业批改、付费课程体系的机构用户。
- 需要企业级权限、审计、多人协作和数据看板的团队用户。

## 4. 用户痛点

### 4.1 学习路径碎片化

Git、GitHub、AI 编程、CI/CD、发布都有大量教程，但大多按技术知识点组织，不按产品经理的真实工作流组织。用户看完后很难判断这些知识如何服务需求管理、研发协作和上线交付。

### 4.1.1 学习资源需要动态调整

12 周计划不绑定固定课程。用户会不断提交自己看过的文章、视频、工具和实践内容，因此产品需要支持资源记录、资源替换和资源推荐，而不是把学习路径写死。

### 4.2 学完不会用

用户容易停留在“看视频、记笔记”，缺少实际任务、产出模板和验收标准。学习行为没有转化为工作资产。

### 4.3 与研发沟通缺少抓手

用户知道要和研发沟通，但不知道在 PR、冲突、发布失败、安全风险、CI 报错等场景下该问什么，也不知道哪些问题需要升级。

### 4.4 AI 使用缺少约束

用户会让 AI 写代码、解释代码或拆需求，但经常缺少目标、上下文、边界、验收标准和输出格式，导致 AI 结果难以复用，甚至扩大风险。

### 4.5 学习成果不可见

用户学完后没有可展示成果。没有 GitHub 仓库、PR 记录、项目复盘、发布地址，也就难以证明自己完成了从产品到交付的能力升级。

## 5. 用户目标

用户使用本产品后，希望达成：

- 知道 Git/GitHub 在研发协作中分别解决什么问题。
- 能看懂并参与 Issue、Branch、Commit、Pull Request、Review、Merge 的流程。
- 遇到常见 Git 问题时知道如何低风险判断状态。
- 能使用 AI 辅助 PRD 拆解、代码解释、PR Review、排障、发布检查和学习复盘。
- 能在 12 周内完成一个小产品闭环。
- 能产出可展示材料：学习仓库、周报、PRD、Issue、PR、发布地址、项目复盘。
- 能持续沉淀个人学习资产，为未来 blog、公开工具、个人品牌和一人公司方向积累素材。

## 6. 业务目标

当前阶段明确作为个人工具，业务目标应聚焦于个人使用价值：

- 降低开始学习全链路技术协作的门槛。
- 提高用户每周持续推进的概率。
- 让每周学习形成明确产出。
- 帮助用户完成最终闭环项目。
- 建立长期个人学习档案和可复用方法论。

半年到一年后，如果学习资产足够稳定，可以顺手考虑公开化：

- blog 内容沉淀。
- 个人工具公开访问。
- 用户留存。
- 模板复制率。
- 课程/训练营包装。
- 社群传播。

## 7. 核心使用场景

### 7.1 每周开始时规划学习

用户打开今日面板，看到当前周主题、完成进度、本周任务和下一步行动。

用户需要回答：

- 这周学什么？
- 为什么学？
- 做到什么程度算完成？
- 今天先做哪一件事？

### 7.2 学习过程中记录和复盘

用户完成任务后勾选进度，并记录笔记。周末导出周报或复盘，放入 GitHub 仓库。

用户需要回答：

- 我本周完成了什么？
- 哪些概念真正能用于工作？
- 我卡在哪里？
- 下周最小行动是什么？

### 7.2.1 提交和推荐学习资源

用户把看过的文章、视频、课程、工具和实践材料提交到工具中，记录资源来源、主题、难度、价值和适用场景。系统根据当前周主题、用户基础和已完成内容，推荐下一批更合适的资源。

用户需要回答：

- 这个资源解决了什么问题？
- 它适合哪个学习阶段？
- 我看完之后产出了什么？
- 下一步应该看更基础、更实战，还是更系统的内容？

### 7.3 遇到 Git 问题时自救

用户进入 Git 急救，搜索“冲突、撤销、PR、同步”等关键词，获得可操作参考。

用户需要回答：

- 当前状态是否安全？
- 先看什么命令？
- 哪些动作风险高？
- 需要向研发确认什么？

### 7.4 用 AI 辅助研发协作

用户进入 AI 提示词库，复制适合场景的模板，用于 PRD 拆解、代码解释、Review、排障、发布检查。

用户需要回答：

- 我该怎么给 AI 上下文？
- 我要限制 AI 不做什么？
- 输出应该是什么格式？
- 如何判断 AI 结果是否可用？

### 7.5 完成闭环项目

用户在第 11-12 周选择一个真实但范围小的工作场景，完成问题定义、需求拆解、AI 协作实现、Git 流程、发布和复盘。

用户需要回答：

- 这个小产品解决谁的什么问题？
- 最小可行范围是什么？
- 怎么验收？
- 怎么发布？
- 最后如何展示？

## 8. 现有能力评估

### 8.1 已具备能力

- 12 周学习计划。
- 当前周自动定位。
- 任务完成状态保存。
- 每周笔记。
- Git 急救卡搜索。
- AI 提示词搜索与复制。
- 闭环项目里程碑。
- JSON 导入导出。
- Markdown 周报导出。
- 前端登录门禁。
- PWA 基础文件。
- 静态部署能力。

### 8.2 主要不足

- 缺少首次使用引导，用户不知道自己适合从哪里开始。
- 任务完成标准还不够强，部分任务缺少可复制产出模板。
- 笔记和复盘能力偏轻，周报价值可以进一步提升。
- Git 急救卡还可以强化风险提示和协作问题。
- AI 提示词库还不能自定义和沉淀个人模板。
- 缺少学习资源提交、评价和推荐能力。
- 手动导入导出适合早期，但长期多端使用成本较高。
- 前端登录门禁无法提供真正安全保护。

## 9. 需求优先级

### P0：必须优先满足

1. 用户每次打开都知道下一步做什么。
2. 每周任务有明确完成定义。
3. 每周学习能沉淀为复盘或周报。
4. Git 急救能帮助用户低风险判断问题。
5. AI 提示词能服务真实研发协作场景。
6. 最终能引导用户完成一个小产品闭环。
7. 登录门禁需要升级为真正能保护学习数据的认证方案。

### P1：增强长期价值

1. 自定义 Git 急救卡。
2. 自定义 AI 提示词。
3. 学习资产库。
4. 闭环项目选题向导。
5. 最近学习提醒。
6. 更清晰的成果展示页。
7. 学习资源库和资源推荐。

### P2：未来扩展

1. 真实账号体系。
2. 云端同步。
3. 客户端应用。
4. 多人学习空间。
5. 训练营/课程化后台。
6. 数据看板与学习分析。

## 10. 成功指标

个人工具阶段建议关注：

- 12 周任务完成率。
- 每周复盘填写率。
- 周报导出次数。
- Git 急救卡使用次数。
- AI 提示词复制次数。
- 学习资源提交数量与再次使用次数。
- 最终闭环项目完成率。
- 是否沉淀至少 3 个可复用资产：PRD、PR 模板、发布检查表、项目复盘。

公开产品阶段可追加：

- 首次访问到完成第一项任务的转化率。
- 第 7 天留存。
- 第 4 周留存。
- 模板复制率。
- 导入导出使用率。
- 用户推荐意愿。

## 11. 风险与约束

### 11.1 产品风险

- 12 周周期较长，用户可能中途放弃。
- 学习内容如果过于技术化，会让产品经理用户产生畏难。
- 如果任务过于抽象，用户会继续停留在“看过但不会用”。
- 如果闭环项目范围过大，用户最后无法完成。
- 如果资源推荐质量不高，会变成另一个收藏夹，而不是学习教练。

### 11.2 技术约束

- 当前基础版本是静态应用，但“真正保护学习数据”需要引入服务端认证或可信托管认证。
- 数据保存在浏览器 localStorage。
- 当前前端登录门禁不具备真正安全性，不能满足长期数据保护目标。
- 短期多端同步先通过浏览器能力和手动导入导出解决。
- 中期如果数据重要性提升，需要引入真实登录、云端同步或客户端。
- 暂不适合保存敏感信息。

### 11.3 运营约束

- 如果未来做公开产品，需要补充内容更新机制。
- 如果做训练营，需要导师反馈、作业批改、班级管理等能力。
- 如果做商业化，需要重新定义用户付费价值。

## 12. 已确认补充

1. 当前阶段目标是自己使用，用于各种学习、记录和实践沉淀；半年或一年后，可能考虑做成 blog 或工具供其他人使用。
2. 当前目标用户是有很浅显开发基础的产品经理，只了解大概开发概念，需要从研发协作链路逐步补齐能力。
3. 12 周计划的配套资源不固定，可以随时调整；用户会提交看过的相关内容，工具需要支持资源记录和合适资源推荐。
4. 最终更想产出个人学习工具；包装成课程或训练营可以是顺手发生的结果，而不是当前主目标。
5. 登录门禁希望真正保护学习数据，因此当前前端门禁只能作为临时方案，后续需要真实认证。
6. 多端同步近期先通过浏览器和导入导出解决，后续可以考虑开发客户端。
7. 项目服务于个人成长目标：打造有竞争力的个体，为一人公司、独立产品、个人品牌或真正意义上的财务自由做铺垫。
8. 产品调性应偏陪伴式学习教练，而不是冷冰冰的严肃工作台。

补充说明：当前文档只能基于本工作区和本次对话中的个人信息进行设计，无法直接读取用户在 ChatGPT 其他对话中的个人资料。后续可以增加“个人画像”模块，由用户主动补充长期目标、能力栈、资源偏好和阶段计划。

## 13. 需求结论

当前最重要的方向是把产品从“学习计划页面”升级为“陪伴式个人学习与产出系统”。

短期不建议优先做复杂后端、多人体系或商业化包装。更值得先补齐：

- 个人画像与首次引导。
- 每周完成定义。
- 每周产出模板。
- 更强复盘能力。
- 学习资源库和资源推荐。
- Git 风险提示。
- 闭环项目选题与交付模板。
- 真实认证方案评估。

只要这些闭环成立，即使早期仍主要依赖浏览器本地能力，也已经具备清晰的个人价值。后续是否公开成 blog、工具、课程或训练营，应等个人使用路径和学习资产足够稳定后再判断。
