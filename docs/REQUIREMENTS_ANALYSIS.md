# 全链路学习助手：需求分析文档

版本：V1.1  
日期：2026-05-24  
项目阶段：基础部署后，进入个人学习工具产品化设计阶段

## 1. 项目背景

当前项目是一个已完成基础部署的 Web 应用，定位为“全链路学习助手”。现阶段它首先服务于王坤（kun）的个人长期学习、记录和能力建设，围绕 Git/GitHub、AI 辅助编程、研发协作、CI/CD、发布和复盘，提供一套可调整的 12 周学习与实践路径。

结合个人画像，用户当前是医疗信息化领域的产品经理/研发管理者，长期从事 HIS、医院管理信息系统、药品管理、费用管理、预约、执行、医技等相关业务。这个工具不只是学习 Git 和 AI 编程，更重要的是帮助用户把医疗信息化复杂业务、产品管理、研发协同、交付治理沉淀成长期竞争力。

从现有产品形态看，它已经具备学习计划、任务勾选、笔记、Git 急救、AI 提示词、闭环项目、多端导入导出和前端登录门禁等能力。

本阶段的核心任务不是继续堆功能，而是先明确：

- 这个产品如何优先服务个人成长。
- 用户为什么需要它。
- 它和普通课程、笔记工具、项目管理工具有什么区别。
- 下一阶段应该优先补齐哪些能力。

## 2. 产品定位

一句话定位：

> 面向医疗信息化产品经理/研发管理者的陪伴式全链路学习教练，帮助用户围绕真实业务、研发协作、AI 工具化和项目交付，持续沉淀知识库、方法论和可复用工具，逐步打造复杂业务领域的个人核心竞争力。

产品关键词：

- 个人成长操作台
- 陪伴式学习教练
- 产品经理技术协作
- 医疗信息化复杂业务
- HIS / 药品 / 费用核心域
- 研发协同与交付治理
- Git/GitHub 入门到协作
- AI 辅助编程与研发管理
- 小产品闭环交付
- 资源推荐与学习记录
- 知识库与方法论沉淀
- 可追踪、可复盘、可展示

产品不应被设计成普通课程站点。它的核心价值不是“展示学习资料”，而是把学习过程变成一个可执行、可追踪、可复盘、可展示的个人成长项目。半年到一年后，产品可以自然外化为 blog、公开工具或个人方法论展示页，但短期所有设计都应优先服务自用效率和长期沉淀。

## 3. 目标用户

### 3.1 当前核心用户

当前最核心用户是王坤本人：医疗信息化领域的产品经理/研发管理者，有较强业务建模、需求分析、产品设计、流程治理、数据结构理解和系统拆解能力；不是纯技术开发，但理解开发、前端、后端、测试、上线等基础概念，需要进一步系统掌握 GitHub 协作、AI 辅助编程、CI/CD 和完整交付链路。

这个产品更像个人学习教练、成长档案和知识生产工作台，而不是通用 SaaS。它服务的长期目标是：先把药品、费用两个核心域吃透，再逐步形成完整医疗信息化认知体系，进一步沉淀为可复用的方法论、知识库和 AI 协作体系，最终具备迁移到其他复杂业务领域的能力，并为未来一人公司、个人品牌、独立工具产品、财务自由等方向积累能力和资产。

### 3.1.1 当前个人能力画像

优势能力：

- 医疗信息化业务理解，尤其是 HIS、药品管理、费用管理、门诊/住院流程、医嘱、收费、执行、发药链路。
- 复杂业务建模、流程拆解、需求分析、产品设计和研发协同。
- 对产品化交付、项目版本、主线版本、分支合并、发版质量和交付治理有真实场景积累。
- 能理解数据表结构、状态流转、参数体系、规则体系和系统边界。

待补能力：

- Git/GitHub 协作链路的系统化掌握。
- AI 辅助编程、排障、代码理解和自动化脚本能力。
- CI/CD、测试、发布、回滚、质量门禁等工程交付能力。
- 把个人经验沉淀为稳定知识库、工具化产品和公开表达资产的能力。

### 3.2 未来可扩展用户

#### 用户 A：在职产品经理

典型特征：

- 参与需求、评审、排期、验收和上线。
- 有很浅显的开发概念，经常和研发协作，但对 Git、PR、CI/CD、发布链路缺乏系统理解。
- 希望自己能更专业地判断研发风险和沟通问题。

核心诉求：

- 看懂研发在做什么。
- 看懂 PR、分支、冲突、发布失败等常见协作问题。
- 能把 PRD 拆成更工程化的任务。
- 会用 AI 辅助需求拆解、代码理解、排障和复盘。

#### 用户 B：研发管理者或技术协作负责人

典型特征：

- 不一定每天写代码，但需要理解研发流程和工程质量。
- 关注团队协作、质量门禁、发布风险、权限和安全。

核心诉求：

- 系统梳理研发协作链路。
- 把 GitHub、PR、CI/CD、发布检查转化为管理动作。
- 建立可复用的协作模板和检查清单。

#### 用户 C：想转向 AI 产品或技术产品的人

典型特征：

- 有产品或业务经验。
- 想补齐技术理解力，避免只停留在概念层。
- 希望产出一个可展示的小作品。

核心诉求：

- 用低门槛路径补齐技术协作能力。
- 通过一个完整小产品证明自己理解 AI 与研发交付。

### 3.3 非目标用户

当前阶段不优先服务：

- 只想系统学习编程语法的零基础编程用户。
- 需要算法、计算机基础、后端架构深度课程的用户。
- 当前就需要班级、助教、作业批改、付费课程体系的机构用户。
- 需要企业级权限、审计、多人协作和数据看板的团队用户。

## 4. 用户痛点

### 4.1 学习路径碎片化

Git、GitHub、AI 编程、CI/CD、发布都有大量教程，但大多按技术知识点组织，不按产品经理的真实工作流组织。用户看完后很难判断这些知识如何服务需求管理、研发协作和上线交付。

### 4.1.1 学习资源需要动态调整

12 周计划不绑定固定课程。用户会不断提交自己看过的文章、视频、工具和实践内容，因此产品需要支持资源记录、资源替换和资源推荐，而不是把学习路径写死。

### 4.2 学完不会用

用户容易停留在“看视频、记笔记”，缺少实际任务、产出模板和验收标准。学习行为没有转化为工作资产。

### 4.3 与研发沟通缺少抓手

用户知道要和研发沟通，但不知道在 PR、冲突、发布失败、安全风险、CI 报错等场景下该问什么，也不知道哪些问题需要升级。

### 4.4 AI 使用缺少约束

用户会让 AI 写代码、解释代码或拆需求，但经常缺少目标、上下文、边界、验收标准和输出格式，导致 AI 结果难以复用，甚至扩大风险。

### 4.5 学习成果不可见

用户学完后没有可展示成果。没有 GitHub 仓库、PR 记录、项目复盘、发布地址，也就难以证明自己完成了从产品到交付的能力升级。

### 4.6 医疗信息化经验沉淀成本高

用户长期面对 HIS、药品、费用、预约、医技、执行、库存、版本治理等复杂业务，但经验容易散落在会议纪要、需求文档、腾讯文档、项目计划、汇报材料和即时沟通中。如果没有统一沉淀机制，个人经验很难升级为稳定方法论、知识库和工具化资产。

### 4.7 项目交付与产品化之间需要持续平衡

医疗信息化项目高度个性化，现场交付压力大，主线产品长期演进又需要稳定规则和抽象能力。用户需要一个工具帮助自己在“解决当前项目问题”和“沉淀可复用产品能力”之间持续做取舍。

## 5. 用户目标

用户使用本产品后，希望达成：

- 知道 Git/GitHub 在研发协作中分别解决什么问题。
- 能看懂并参与 Issue、Branch、Commit、Pull Request、Review、Merge 的流程。
- 遇到常见 Git 问题时知道如何低风险判断状态。
- 能使用 AI 辅助 PRD 拆解、代码解释、PR Review、排障、发布检查和学习复盘。
- 能在 12 周内完成一个小产品闭环。
- 能产出可展示材料：学习仓库、周报、PRD、Issue、PR、发布地址、项目复盘。
- 能持续沉淀个人学习资产，为未来 blog、公开工具、个人品牌和一人公司方向积累素材。
- 能把 HIS、药品、费用等真实业务经验沉淀为结构化知识库、领域模型、流程模板、验收清单和研发协同规范。
- 能形成 AI 协作体系，让 AI 成为产品参谋、研发协同助手和系统设计搭档。

## 6. 业务目标

当前阶段明确作为个人工具，业务目标应聚焦于个人使用价值：

- 降低开始学习全链路技术协作的门槛。
- 提高用户每周持续推进的概率。
- 让每周学习形成明确产出。
- 帮助用户完成最终闭环项目。
- 建立长期个人学习档案和可复用方法论。
- 支撑医疗信息化知识库和 AI 协作体系持续积累。
- 把工作中的 PRD、详细设计、流程图、汇报材料、会议纪要、方案对比、管理制度等沉淀为个人资产。

半年到一年后，如果学习资产足够稳定，可以顺手考虑公开化：

- blog 内容沉淀。
- 个人工具公开访问。
- 用户留存。
- 模板复制率。
- 课程/训练营包装。
- 社群传播。

## 7. 核心使用场景

### 7.1 每周开始时规划学习

用户打开今日面板，看到当前周主题、完成进度、本周任务和下一步行动。

用户需要回答：

- 这周学什么？
- 为什么学？
- 做到什么程度算完成？
- 今天先做哪一件事？

### 7.2 学习过程中记录和复盘

用户完成任务后勾选进度，并记录笔记。周末导出周报或复盘，放入 GitHub 仓库。

用户需要回答：

- 我本周完成了什么？
- 哪些概念真正能用于工作？
- 我卡在哪里？
- 下周最小行动是什么？

### 7.2.1 提交和推荐学习资源

用户把看过的文章、视频、课程、工具和实践材料提交到工具中，记录资源来源、主题、难度、价值和适用场景。系统根据当前周主题、用户基础和已完成内容，推荐下一批更合适的资源。

用户需要回答：

- 这个资源解决了什么问题？
- 它适合哪个学习阶段？
- 我看完之后产出了什么？
- 下一步应该看更基础、更实战，还是更系统的内容？

### 7.3 遇到 Git 问题时自救

用户进入 Git 急救，搜索“冲突、撤销、PR、同步”等关键词，获得可操作参考。

用户需要回答：

- 当前状态是否安全？
- 先看什么命令？
- 哪些动作风险高？
- 需要向研发确认什么？

### 7.4 用 AI 辅助研发协作

用户进入 AI 提示词库，复制适合场景的模板，用于 PRD 拆解、代码解释、Review、排障、发布检查。

用户需要回答：

- 我该怎么给 AI 上下文？
- 我要限制 AI 不做什么？
- 输出应该是什么格式？
- 如何判断 AI 结果是否可用？

### 7.5 完成闭环项目

用户在第 11-12 周选择一个真实但范围小的工作场景，完成问题定义、需求拆解、AI 协作实现、Git 流程、发布和复盘。

用户需要回答：

- 这个小产品解决谁的什么问题？
- 最小可行范围是什么？
- 怎么验收？
- 怎么发布？
- 最后如何展示？

### 7.6 沉淀医疗信息化知识库

用户在日常工作中处理药品、费用、预约、执行、医技、版本治理等复杂问题，将关键经验整理为领域卡片、流程图、验收清单、参数说明、数据结构说明和协作规范。

用户需要回答：

- 这个问题属于哪个业务域？
- 它背后的规则、状态、数据对象是什么？
- 哪些是项目个性化，哪些可以产品化复用？
- 后续遇到类似问题时如何快速复用？

### 7.7 辅助研发协同和管理汇报

用户需要把复杂项目状态转成领导能看懂、研发能执行、测试能验收的材料，包括 PRD、详细设计、需求分析、方案对比、版本治理方案、汇报材料和会议纪要。

用户需要回答：

- 这份材料给谁看？
- 要推动什么决策？
- 当前结论是什么？
- 风险和下一步动作是什么？
- 怎么验收？

## 8. 现有能力评估

### 8.1 已具备能力

- 12 周学习计划。
- 当前周自动定位。
- 任务完成状态保存。
- 每周笔记。
- Git 急救卡搜索。
- AI 提示词搜索与复制。
- 闭环项目里程碑。
- JSON 导入导出。
- Markdown 周报导出。
- 前端登录门禁。
- PWA 基础文件。
- 静态部署能力。

### 8.2 主要不足

- 缺少首次使用引导，用户不知道自己适合从哪里开始。
- 任务完成标准还不够强，部分任务缺少可复制产出模板。
- 笔记和复盘能力偏轻，周报价值可以进一步提升。
- Git 急救卡还可以强化风险提示和协作问题。
- AI 提示词库还不能自定义和沉淀个人模板。
- 缺少学习资源提交、评价和推荐能力。
- 缺少医疗信息化领域知识库能力，无法承载药品、费用、预约、执行、医技等业务资产。
- 缺少“项目经验转方法论”的结构化沉淀机制。
- 手动导入导出适合早期，但长期多端使用成本较高。
- 前端登录门禁无法提供真正安全保护。

## 9. 需求优先级

3. 每周学习能沉淀为复盘或周报。
4. Git 急救能帮助用户低风险判断问题。
5. AI 提示词能服务真实研发协作场景。
6. 最终能引导用户完成一个小产品闭环。
7. 登录门禁需要升级为真正能保护学习数据的认证方案。
8. 支持沉淀医疗信息化领域知识、流程模板、验收清单和研发协同规范。

### P1：增强长期价值

1. 自定义 Git 急救卡。
2. 自定义 AI 提示词。
3. 学习资产库。
4. 闭环项目选题向导。
5. 最近学习提醒。
6. 更清晰的成果展示页。
7. 学习资源库和资源推荐。
8. 医疗信息化领域知识库。
9. 汇报材料、设计文档和研发协同模板库。

### P2：未来扩展

1. 真实账号体系。
2. 云端同步。
3. 客户端应用。
4. 多人学习空间。
5. 训练营/课程化后台。
6. 数据看板与学习分析。

## 10. 成功指标

个人工具阶段建议关注：

- 12 周任务完成率。
- 每周复盘填写率。
- 周报导出次数。
- Git 急救卡使用次数。
- AI 提示词复制次数。
- 学习资源提交数量与再次使用次数。
- 医疗信息化知识卡片沉淀数量。
- 业务域复盘数量，例如药品、费用、门诊、住院、医技、预约。
- 最终闭环项目完成率。
- 是否沉淀至少 3 个可复用资产：PRD、PR 模板、发布检查表、项目复盘。
- 是否沉淀至少 3 个领域资产：业务流程、数据模型、验收清单、参数规则、版本治理方案。

公开产品阶段可追加：

- 首次访问到完成第一项任务的转化率。
- 第 7 天留存。
- 第 4 周留存。
- 模板复制率。
- 导入导出使用率。
- 用户推荐意愿。

## 11. 风险与约束

### 11.1 产品风险

- 12 周周期较长，用户可能中途放弃。
- 学习内容如果过于技术化，会让产品经理用户产生畏难。
- 如果任务过于抽象，用户会继续停留在“看过但不会用”。
- 如果闭环项目范围过大，用户最后无法完成。
- 如果资源推荐质量不高，会变成另一个收藏夹，而不是学习教练。
- 如果把医疗信息化知识库做得过大，容易变成重型知识管理系统，反而拖慢学习闭环。
- 如果过早公开化，可能会受限于工作内容敏感性、客户信息和公司内部资料边界。

### 11.2 技术约束

- 当前基础版本是静态应用，但“真正保护学习数据”需要引入服务端认证或可信托管认证。
- 数据保存在浏览器 localStorage。
- 当前前端登录门禁不具备真正安全性，不能满足长期数据保护目标。
- 短期多端同步先通过浏览器能力和手动导入导出解决。
- 中期如果数据重要性提升，需要引入真实登录、云端同步或客户端。
- 暂不适合保存敏感信息。

### 11.3 运营约束

- 如果未来做公开产品，需要补充内容更新机制。
