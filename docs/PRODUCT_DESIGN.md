# 全链路学习助手：产品设计文档

版本：V1.1  
日期：2026-05-24  
关联文档：`docs/REQUIREMENTS_ANALYSIS.md`

## 1. 设计目标

本产品设计文档基于需求分析结论，目标是定义“全链路学习助手”下一阶段的产品结构、页面设计、功能模块、数据对象和迭代优先级。当前产品首先服务个人长期学习和记录，未来半年到一年后再判断是否外化为 blog、公开工具、课程或训练营。

设计原则：

- 让用户每次打开都知道下一步做什么。
- 让学习行为形成可见产出。
- 让产品经理用研发协作语言理解技术链路。
- 让 AI 使用有上下文、有约束、有验收。
- 用陪伴式学习教练的语气帮助用户持续推进，而不是只做冷冰冰的任务面板。
- 在浏览器可承载的范围内优先完成个人闭环，同时为真实认证、同步和客户端预留升级路径。

## 3. 用户主流程

### 3.1 首次使用流程

目标：帮助用户理解产品价值，建立个人画像，并选择适合自己的起点。

流程：

1. 用户登录。
2. 进入首次引导。
3. 选择身份：
   - 浅技术基础产品经理
   - 研发管理能力补齐
   - AI 产品/技术产品转型
   - 一人公司/独立产品准备
4. 选择基础：
   - 完全不了解 Git
   - 用过 GitHub 网页
   - 能看懂一点代码
   - 已参与过 PR 或发布
5. 选择目标：
   - 看懂研发协作
   - 用 AI 辅助写小工具
   - 完成一个可展示项目
   - 提升研发管理能力
   - 沉淀个人方法论与 blog 素材
   - 为一人公司和独立产品积累能力资产
6. 系统生成建议：
   - 推荐起始周
   - 每周节奏
   - 首个行动
   - 最终项目方向建议
   - 推荐学习资源类型

MVP 处理方式：

- 不必真的做复杂算法。
- 可以用简单规则映射推荐。
- 早期数据仍可保存在 localStorage，但要明确提示当前保护边界。
- 后续真实保护学习数据时，优先接入服务端认证或可信托管认证。

### 3.2 日常学习流程

1. 用户打开今日面板。
2. 查看当前周主题与下一步行动。
3. 进入 12 周计划完成任务。
4. 使用模板完成本周产出。
5. 记录笔记和复盘。
6. 周末导出周报。
7. 将有价值内容沉淀为 blog 素材、方法论片段或个人资产。

### 3.2.1 学习资源提交与推荐流程

1. 用户看到一篇文章、视频、课程、工具或案例。
2. 进入学习资源模块，提交标题、链接、主题、难度、来源和个人评价。
3. 选择关联周次或能力标签。
4. 系统根据当前学习阶段、已完成任务和资源标签，推荐下一批资源。
5. 用户完成资源学习后，记录收获和可复用产出。

### 3.3 问题自救流程

1. 用户遇到 Git 或协作问题。
2. 进入 Git 急救。
3. 搜索关键词。
4. 查看场景说明、风险等级、推荐命令、协作问题。
5. 根据提示先判断状态，再执行低风险动作。

### 3.4 AI 协作流程

1. 用户进入 AI 提示词。
2. 按场景搜索或筛选。
3. 复制模板。
4. 替换上下文。
5. 将结果沉淀为 Issue、PRD、Review、检查表或复盘。

### 3.5 闭环项目流程

1. 用户进入闭环项目。
2. 完成选题评估。
3. 生成 1 页 PRD。
4. 拆分 3-5 个 Issue。
5. 用 AI 辅助实现核心功能。
6. 完成 PR 和自我 Review。
7. 发布可访问版本。
8. 写项目复盘。
9. 将项目复盘拆分为公开 blog、作品集或未来工具介绍素材。

## 4. 页面设计

### 4.1 今日面板

定位：用户的学习驾驶舱。

现有内容：

- 总体完成度
- 当前主题
- 本周投入
- 最终产出
- 下一步行动
- 本周任务
- 周节奏

建议增强：

1. 今日只做一件事
   - 从当前周未完成任务中选出优先任务。
   - 文案应明确到动作，例如“创建学习仓库并写 README”。

2. 最近复盘摘要
   - 展示最近一次填写的复盘。
   - 帮助用户接上上次学习状态。

3. 本周产出入口
   - 显示本周应产出的文档或材料。
   - 可点击进入模板。

4. 学习节奏状态
   - 例如“本周已完成 2/4 项”。
   - 如果用户落后，提示调整而不是制造压力。

5. 教练式提醒
   - 用温和、具体的方式提示用户继续推进。
   - 例如“今天不用补完全部，先把这条资源的收获写成 3 句话。”

核心字段：

- currentWeek
- overallProgress
- nextAction
- weeklyOutput
- recentReview
- weeklyTaskProgress
- coachMessage

### 4.2 12 周计划

定位：完整路线和任务执行区。

现有内容：

- 周主题
- 学习负载
- 目标产出
- 视频/材料
- 任务列表
- 笔记

建议增强：

1. 完成定义
   - 每周都要说明“做到什么算完成”。
   - 示例：第 3 周完成定义是“创建一个 Issue，基于该 Issue 提交一次 PR，并写清楚 PR 描述和验证方式”。

2. 本周产出模板
   - 每周提供一个可复制模板。
   - 例如概念地图、PRD、Issue、PR 描述、发布检查表、复盘。

3. 工作场景应用
   - 说明本周能力如何用于真实工作。
   - 例如“看 PR 时不只看页面效果，还要看影响范围、验证方式和回滚风险”。

4. 常见误区
   - 降低学习风险。
   - 例如“不要在不理解状态时直接复制危险命令”。

5. 推荐资源
   - 展示当前周可选资源，不强绑定固定课程。
   - 支持用户标记“已看过”“以后再看”“不适合我”。

6. 资源提交入口
   - 用户可以把看过的内容提交到当前周。
   - 提交后进入学习资源库，作为后续推荐依据。

### 4.3 Git 急救

定位：低风险问题判断和协作表达工具。

现有内容：

- 搜索
- 场景卡片
- why 说明
- command 命令

建议卡片结构：

- 标题
- 关键词
- 适用场景
- 风险等级：低 / 中 / 高
- 先检查什么
- 推荐动作
- 不建议做什么
- 需要向研发确认的问题

示例：

标题：我遇到了合并冲突  
风险等级：中  
先检查：

```bash
git status
```

推荐动作：

- 打开冲突文件。
- 找到冲突标记。
- 确认保留哪部分内容。
- 处理后提交。

需要向研发确认：

- 两边改动分别对应什么需求？
- 哪个版本符合当前验收标准？
- 是否需要补充测试？

### 4.4 AI 提示词

定位：产品经理的 AI 协作模板库。

现有内容：

- 搜索
- 场景说明
- 提示词模板
- 复制

建议分类：

- 需求拆解
- 代码解释
- Git 排障
- PR Review
- 发布检查
- 研发排期
- 学习复盘

建议增强：

1. 模板质量提示
   - 每个模板提醒用户补齐目标、上下文、约束、验收标准。

2. 最近使用
   - 显示最近复制过的提示词。

3. 自定义模板
   - 用户可以新增自己的提示词。
   - 保存到 localStorage。

4. 输出用途
   - 每个模板说明结果可以变成什么资产，例如 Issue、PRD、检查表、复盘。

### 4.5 闭环项目

定位：最终成果生产区，也是个人竞争力资产沉淀区。

现有内容：

- 项目建议
- 里程碑
- 验收标准

建议增强：

1. 选题向导
   - 问题：这个工具给谁用？
   - 频率：这个问题每周出现几次？
   - 范围：是否能在 1-2 个页面内完成？
   - 数据：是否需要后端？
   - 发布：是否能用 GitHub Pages 或静态部署？

2. PRD 模板
   - 用户
   - 问题
   - 目标
   - 非目标
   - 核心流程
   - 验收标准
   - 风险

3. Issue 拆解模板
   - 标题
   - 背景
   - 任务描述
   - 验收标准
   - 依赖
   - 风险

4. 发布检查表
   - 功能验收
   - 样式和移动端
   - 数据保存
   - 权限和密钥
   - 回滚方案
   - 发布地址

5. 项目复盘模板
   - 目标
   - 做了什么
   - 关键取舍
   - AI 如何参与
   - Git 流程如何使用
   - 遇到的问题
   - 下一步

6. 公开化素材
   - blog 标题草稿
   - 工具介绍文案
   - 作品集摘要
   - 可以公开的截图或演示链接

### 4.6 学习资源

定位：个人学习资源库和推荐引擎。

核心目标：

- 记录用户已经看过、正在看、准备看的资源。
- 避免资源收藏后失联。
- 将资源与 12 周计划、能力标签和实际产出关联起来。
- 根据当前阶段推荐更合适的下一步资源。

资源类型：

- 文章
- 视频
- 课程
- 官方文档
- GitHub 仓库
- 工具
- 案例
- 个人笔记

资源字段：

- 标题
- 链接
- 类型
- 来源
- 关联周次
- 能力标签
- 难度：入门 / 进阶 / 实战 / 查漏补缺
- 状态：待读 / 已读 / 复看 / 不适合
- 价值评分
- 学习收获
- 产出链接

推荐规则 MVP：

- 当前周优先推荐同周次、入门或实战资源。
- 如果某周任务完成率低，优先推荐基础解释型资源。
- 如果某周任务已完成，推荐案例、项目或复盘型资源。
- 用户标记“不适合”的资源不再推荐。

### 4.7 多端与安全

定位：数据可带走、数据可保护、未来可升级为客户端。

现有内容：

- 直接打开 index.html
- 导入导出 JSON
- GitHub Pages 推荐
- 数据边界说明

建议增强：

1. 数据说明更明确
   - 哪些数据保存在本地。
   - 哪些数据会导出。
   - 哪些数据不应填写，例如密钥、Token、真实敏感项目信息。

2. 同步方案分级
   - 当前：浏览器本地存储 + 手动导入导出。
   - 近期：充分利用浏览器能力，优化备份和恢复体验。
   - 中期：GitHub Gist、Supabase 或其他轻量云同步。
   - 长期：客户端应用，支持更稳定的本地数据和多端体验。

3. 备份提醒
   - 提醒用户定期导出 JSON。

4. 真实认证
   - 当前前端门禁只能作为临时防误入。
   - 如果要真正保护学习数据，应优先评估 Cloudflare Access、Supabase Auth 或自建服务端认证。
   - 在真实认证完成前，不建议保存敏感学习数据、密钥、Token 或未公开项目细节。

## 5. 功能模块设计

### 5.1 个人画像与首次引导模块

功能：

- 收集用户身份、基础、目标。
- 生成推荐学习起点。
- 保存用户画像。
- 记录长期目标，例如一人公司、独立产品、个人品牌、财务自由。
- 为资源推荐和教练式提醒提供上下文。

数据结构示例：

```json
{
  "profile": {
    "role": "product_manager",
    "level": "github_beginner",
    "goal": "complete_capstone",
    "longTermGoal": "solo_company",
    "tone": "coach",
    "createdAt": "2026-05-24"
  }
}
```

### 5.2 每周产出模板模块

功能：

- 每周绑定一个或多个模板。
- 支持复制模板。
- 支持用户填写本周产出链接或文本。

数据结构示例：

```json
{
  "weeklyOutputs": {
    "w3": {
      "templateId": "pr-description",
      "content": "",
      "link": ""
    }
  }
}
```

### 5.3 复盘模块

功能：

- 每周填写结构化复盘。
- 周报导出时包含复盘内容。
- 今日面板显示最近复盘摘要。

字段：

- 本周完成
- 最有价值概念
- 工作应用
- 卡点
- 下周行动
- 需要问研发的问题

### 5.4 自定义卡片模块

功能：

- 新增 Git 急救卡。
- 新增 AI 提示词。
- 编辑或删除用户自定义卡片。
- 导入导出包含自定义卡片。

注意：

- 删除功能要谨慎设计，优先做单条删除确认。
- 不做批量删除。

### 5.5 学习资产库模块

功能：

- 汇总用户沉淀的成果。
- 支持填写外部链接。
- 按类型展示。

资产类型：

- README
- 概念地图
- PRD
- Issue
- PR
- 发布地址
- 周报
- 项目复盘
- blog 草稿
- 工具发布页

### 5.6 学习资源模块

功能：

- 新增资源。
- 编辑资源状态。
- 关联周次和能力标签。
- 记录学习收获。
- 基于当前周和用户状态推荐资源。

数据结构示例：

```json
{
  "resources": [
    {
      "id": "r1",
      "title": "GitHub Pull Request 入门",
      "url": "https://example.com",
      "type": "article",
      "source": "manual",
      "weekId": "w3",
      "tags": ["PR", "review"],
      "level": "beginner",
      "status": "read",
      "rating": 4,
      "takeaway": "PR 是协作契约，不只是代码上传。",
      "outputLink": ""
    }
  ]
}
```

### 5.7 认证与数据保护模块

功能：

- 短期保留现有登录门禁，但明确标注保护边界。
- 中期接入真实认证方案。
- 为云同步或客户端预留用户 ID 和数据版本。

候选方案：

- Cloudflare Access：适合先保护入口，改造成本低。
- Supabase Auth：适合账号、数据同步和未来扩展。
- 自建后端认证：控制力强，但维护成本更高。

## 6. 数据保存设计

当前继续使用 localStorage，适合早期浏览器自用版本。由于用户明确希望真正保护学习数据，后续需要将认证和数据保存从“前端门禁 + 本地存储”升级为“真实认证 + 可控同步”。

建议扩展 state：

```json
{
  "done": {},
  "notes": {},
  "milestones": {},
  "expanded": {},
  "profile": {},
  "reviews": {},
  "weeklyOutputs": {},
  "customGitCards": [],
  "customPromptCards": [],
  "assets": [],
  "resources": [],
  "authMeta": {}
}
```

数据原则：

- 不保存敏感密钥。
- 导入时校验 JSON 结构。
- 导出时包含版本号。
- 后续升级数据结构时保留兼容逻辑。
- 在真实认证完成前，明确提示用户不要记录敏感项目资料。
- 为后续客户端或云同步保留数据版本字段。

## 7. 交互设计原则

1. 任务状态明确
   - 未开始、进行中、已完成可以逐步扩展。
   - 当前版本至少保留勾选完成。

2. 复制动作要有反馈
   - 提示“已复制”。
   - 避免用户不确定操作是否成功。

3. 高风险内容要提前提醒
   - Git 撤销、重置、密钥、Token、部署等场景需要风险说明。

4. 文案要面向产品经理
   - 少讲抽象技术术语，多讲协作意义、影响范围和验收方式。

5. 不制造学习焦虑
   - 延迟、未完成时提供调整建议，而不是单纯显示失败。

6. 陪伴式表达
   - 文案要像一个清醒、可靠、能陪用户推进的学习教练。
   - 多给具体下一步，少给空泛鼓励。
   - 允许用户调整计划，而不是把计划当成压力源。

## 8. 视觉与信息表达建议

当前产品应是“陪伴式学习教练 + 个人工作台”的结合。视觉继续保持克制、清晰、偏工具化，但文案和状态反馈应更有陪伴感。

建议：

- 保持左侧导航和右侧主内容结构。
- 卡片用于任务、模板、急救场景，不要过度装饰。
- 风险等级可以用颜色和文字结合表达。
- 主要操作按钮保持少而明确。
- 首页不做营销式大 Hero，直接进入学习操作台。
- 适当使用“今天先做这个”“这周先别贪多”“这个资源值得复看”等教练式短句。

## 9. 迭代路线

### 9.1 V1.1：补齐个人学习闭环

目标：让每周学习都有明确产出。

范围：

- 首次使用引导。
- 每周完成定义。
- 每周产出模板。
- 结构化复盘。

### 9.2 V1.2：学习资源库与推荐

### 9.4 V1.4：闭环项目与资产库

- Issue 拆解模板。
- 发布检查表。

- 学习资产库。

### 9.5 V1.5：真实认证与数据保护

触发条件：

- 用户希望真正保护学习数据。
- 开始保存更长期、更个人化的学习档案。
- 公网访问频率提高。

- Cloudflare Access：保护访问入口。
- Supabase Auth + Database：正式账号和数据同步。

## 10. 验收标准

### 10.1 V1.1 验收标准

- 新用户首次进入可以完成身份、基础、目标选择。
- 用户可以记录长期目标和产品调性偏好。
- 系统能显示推荐起点和首个行动。
- 每周至少有一个明确完成定义。
- 每周至少有一个可复制产出模板。
- 用户可以填写结构化复盘。
- Markdown 周报包含任务完成情况、笔记和复盘。
- 今日面板能显示一条教练式下一步提醒。

### 10.2 V1.2 验收标准

- 用户可以新增学习资源。
- 用户可以为资源设置类型、周次、标签、难度和状态。
- 当前周可以展示推荐资源。
- 周报可以包含本周新增或完成的资源。

### 10.3 V1.3 验收标准

- Git 急救卡显示风险等级。
- Git 急救卡包含先检查什么、推荐动作、需要确认的问题。
- AI 提示词支持分类筛选。
- 用户复制提示词后有明确反馈。
- 用户可以新增至少一条自定义提示词。

### 10.4 V1.4 验收标准

- 用户可以通过选题向导确定闭环项目。
- 用户可以生成或复制 PRD、Issue、发布检查、复盘模板。
- 用户可以在资产库记录至少 5 类成果。
- 最终项目页面能清晰展示完成进度和待补材料。
- 用户可以将项目复盘沉淀为 blog 或公开工具素材。

### 10.5 V1.5 验收标准

- 文档和界面明确说明当前登录门禁的安全边界。
- 完成真实认证方案选型。
- 至少完成一种真实访问保护或认证方案的技术验证。
- 私密数据和未来可公开数据有明确区分。

## 11. 暂不做事项

当前阶段建议暂不做：

- 完整复杂后端系统。
- 多人协作后台。
- 付费系统。
- 班级和作业批改。
- 企业级权限审计。
- 大而全的课程内容管理系统。

原因：

- 当前产品价值还需要先通过个人学习闭环验证。
- 复杂系统会显著增加维护成本。
- 用户真正需要的是长期记录、资源推荐、产出模板、复盘机制和数据保护，而不是过早平台化。

## 12. 产品设计结论

下一阶段最建议优先做 V1.1：补齐个人学习闭环。

具体来说：

- 增加个人画像和首次引导，帮助用户选择起点。
- 给每周增加完成定义和产出模板。
- 强化复盘，让周报变成真正的学习资产。
- 增加教练式提醒，让工具更像陪伴式学习教练。
- 接着推进学习资源库和资源推荐。
- 评估真实认证方案，逐步替代当前前端门禁。

当用户能稳定完成每周任务，并最终交付一个可展示小产品后，再判断是否进入同步、客户端、公开 blog、公开工具或课程/训练营包装。
